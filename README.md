# 🚰 Faucet App

**[→ Live site](https://aleksandralukic.github.io/faucet-app/)**

A live status dashboard for blockchain testnet faucets. 36 faucets across 30+
networks, health-checked every day, so you find out a faucet is dead here
rather than halfway through a test run.

- **Static site** — vanilla HTML/CSS/JS, no build step, no dependencies.
- **Daily cron** — a GitHub Actions job checks every faucet and commits the results.
- **Status history** — each faucet keeps its last 30 check results, rendered as a sparkline.

Built by a QA engineer who got tired of discovering broken faucets the hard way.

## Layout

```
data/faucets.json    the faucet list — the file you edit to add or change entries
data/status.json     generated by the checker; do not hand-edit
scripts/check_faucets.py   the daily checker (Python 3 stdlib only)
assets/              frontend JS + CSS
.github/workflows/daily-check.yml   the cron job
```

## Running locally

```bash
python3 scripts/check_faucets.py   # writes data/status.json
python3 -m http.server 8000        # then open http://localhost:8000
```

The page needs to be served over HTTP — opening `index.html` via `file://` will
fail because `fetch` can't read local files under that scheme.

## Adding a faucet

**Contributions are very welcome** — especially replacements for faucets the
dashboard shows as dead. Fork, edit `data/faucets.json`, open a PR. There's no
build step and no test suite to run; if the JSON parses, CI will check the new
entry on the next run and the result shows up on the site automatically.

Equally useful without writing any code: if you spot a faucet marked `down`
that actually works in a browser (or one marked `up` that's quietly dry),
[open an issue](https://github.com/aleksandralukic/faucet-app/issues) — the
checker can't attempt a real claim, so human reports are the only way to catch
that class of failure.

Append an object to `data/faucets.json`:

```json
{
  "id": "unique-slug",
  "currency": "TOKEN",
  "name": "Human readable name",
  "url": "https://...",
  "network": "Which testnet",
  "requiresWallet": false,
  "requiresCaptcha": true,
  "notes": "Anything a QA engineer needs to know to actually claim."
}
```

`id` must be unique and stable — it's the key that status history is tracked
against. Renaming an `id` resets that faucet's history.

### Optional `check` block

Some entries aren't plain web pages. Override the check behaviour:

```json
"check": {
  "okStatus": [200, 400],     // extra status codes that still mean "healthy"
  "skipContentScan": true,    // don't scan the body for failure keywords
  "mode": "manual",           // don't check at all; needs human verification
  "reason": "Shown in the UI to explain why"
}
```

Already used by:

- **Stellar Friendbot** — a bare GET returns `400` asking for `addr`; that's healthy.
- **Hiro Stacks API** — POST-only endpoint, so any response proves reachability.
- **Westend Matrix Faucet** — the faucet is a Matrix room, not a web page. Marked `manual`.

## What the status actually means

| Status | Meaning |
| --- | --- |
| `up` | Responded with an expected status code and no failure keywords in the body. |
| `degraded` | Responded, but the page mentions maintenance / insufficient funds / bot protection, or returned `403`. |
| `down` | Request failed, timed out, returned an unexpected status, or the page says it's gone or dry. |
| `manual` | Can't be checked automatically. |
| `unknown` | No check has run for this entry yet. |

**A green light does not mean the faucet has funds.** The checker makes a single
`GET` request — it never attempts a claim. It can tell you a faucet's site is
gone; it can't tell you the tap is dry behind a working page.

Two known limits of the content heuristics:

1. **JS-rendered shells.** Many faucets are SPAs that return an near-empty HTML
   shell to a plain `GET`. These are reported `up` with a `js-rendered-shell`
   signal, meaning the keyword scan proved nothing. Upgrading to Playwright
   would fix this at the cost of a much heavier CI job.
2. **Bot protection.** Cloudflare-style interstitials are caught as `degraded`
   rather than `down`, since the faucet itself is probably fine.

Faucets sharing a URL (the MATIC and POL Polygon entries) are fetched once and
the result is shared, so the job doesn't hit operators twice per run.

## The cron job

Runs at 06:15 UTC daily, and on every push to `main`. It needs
`contents: write` permission to commit `data/status.json` back to the repo —
check **Settings → Actions → General → Workflow permissions** is set to
"Read and write permissions" if the push step fails.

Trigger it by hand from the Actions tab via **Run workflow**.

Note that GitHub disables scheduled workflows in repositories with no activity
for 60 days.
